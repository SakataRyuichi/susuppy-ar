# 08. マーカー姿勢制御の設計方針

## 背景

現状の AR 表示では、`水平/垂直` の見え方が期待より曖昧になるケースがある。  
この課題は、以下 2 種類を分けて扱う必要がある。

1. マーカーに対して相対的に水平/垂直を保つ（マーカー基準）
2. 実世界の重力に対して水平/垂直を保つ（重力基準）

## 要件整理

- 現フェーズで優先するのは **マーカー基準** の安定化
- UI からは `水平/垂直`、`動画/3D` を切り替え可能にする
- 既存構成（AR.js + Three.js）を維持し、責務分離を崩さない

## 方針

### Phase 1: マーカー基準の安定化（今回）

- AR.js の `ArMarkerControls` を継続利用
- `changeMatrixMode` は `modelViewMatrix` を採用
  - マーカー姿勢を `markerRoot` 直下で扱いやすくする
- 平滑化パラメータを明示設定し、揺れを抑える
  - `smooth: true`
  - `smoothCount: 8`
  - `smoothTolerance: 0.005`
  - `smoothThreshold: 3`

### Phase 2: マーカー姿勢拘束の厳密化

- `水平/垂直` を固定角度ではなく、マーカー座標系の軸で拘束
- 実装候補:
  - ローカル軸ベースの回転合成
  - クォータニオンによる姿勢補間

### Phase 3: 重力基準モード（必要時）

- `world-gravity` モードを新設
- IMU の重力ベクトルとビジョン姿勢を融合して水平/垂直を定義
- ここは別要件として切り分ける（現フェーズ外）

## 採用理由

- まずは現在の技術スタック内で改善可能な箇所を優先できる
- マーカー基準と重力基準を分離することで、要求の曖昧さを減らせる
- 段階導入により、挙動変化の原因を追跡しやすい

## 非採用案

- いきなり ArUco / AprilTag へ全面移行
  - 移行コストが高く、既存機能への影響範囲が大きい
- いきなり IMU 融合を導入
  - 実装難度と検証難度が高く、フェーズ分割しないとリスクが高い
